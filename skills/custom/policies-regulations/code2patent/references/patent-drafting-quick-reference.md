# 发明专利起草速查卡

> 给 agent 用的短版规则卡。
> 只保留最容易写偏、但最影响 `code2patent` 输出质量的规则。

## 一、先看什么

进入起草层时，默认按这个顺序：

1. `03-方案代码证据映射-<方案名>.md`
2. `04-技术交底书-<方案名>.md`
3. 本速查卡
4. `references/patent-drafting-spec.md`
5. 再动笔写 `04A / 04B / 05 / 05A`

## 二、8 条硬规则

### 1. 权利要求必须有说明书支撑

- 权利要求不能脱离说明书公开内容单独漂移。
- 如果说明书没写清，权利要求就不要先写大。

### 2. 独立权利要求只写必要技术特征

- 只保留“为解决技术问题所必需”的特征。
- 参数、优选条件、缓存、回退、异常处理等，通常先放从属权利要求或实施例。

### 3. 从属权利要求只写附加技术特征

- 不要重复独立权利要求已经写过的全部内容。
- 默认按三层挂接：
  - 核心增强特征
  - 优选参数 / 条件
  - 工程细节

### 4. 先稳 claim tree，再写完整初稿

- 先写 `04A-权利要求布局卡`
- 再写 `04B-权利要求-代码证据矩阵`
- 最后写 `05-发明专利初稿`

不要跳过 L2.5 直接硬写初稿。

### 5. 摘要只做压缩，不做营销

- 写名称、技术领域、技术问题、技术方案要点、主要用途。
- 控制在 300 字内。
- 不写宣传性表达，不写夸张效果。

### 6. 计算机程序相关发明要区分方法口径和产品口径

- 方法口径：写处理步骤。
- 产品口径：可按装置、计算机可读存储介质、计算机程序产品布局。
- 不要只写“某程序实现某功能”这种抽象功能描述。

### 7. A / B / C 级证据进入位置不同

- A 级：可进入独立权利要求骨架。
- B 级：优先进入从属权利要求或优选实施例，并标“基于实现归纳”。
- C 级：只能进待确认事项或替代实施例。

### 8. 第三方能力不直接写成申请人创新

- 模型、SDK、中间件、向量库、云服务的固有能力，不直接主张。
- 只主张本项目对其的组织、调度、适配、回退、缓存、状态管理等自研机制。

## 三、计算机程序相关发明的默认保护主题

默认先并行准备四类草稿，供代理师择优整合：

1. 方法
2. 系统 / 装置
3. 计算机可读存储介质
4. 计算机程序产品

注意：这里是**内部布局草稿**，不等于正式申请文件必须全部同时保留。

## 四、下笔前自问 5 个问题

1. 这份方案真正解决的技术问题是什么？
2. 主独立权利要求里有没有混入“非必要特征”？
3. 说明书是否足以支撑主独立权利要求？
4. 是否把 B / C 级证据写实了？
5. 是否把第三方固有能力误写成了申请人创新？

## 五、最常见的 4 个错误

1. 先写大独权，再补说明书，导致支持不足。
2. 把工程优化细节全塞进独立权利要求，导致主权利要求过窄。
3. 把“程序功能”当成“技术方案”，没有写清技术手段。
4. 把第三方平台能力直接当成申请人的核心创新。

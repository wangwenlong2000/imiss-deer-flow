# 代码抽取规范

> 本规范用于将候选专利方案映射到代码中的真实实现，并形成可供研发人员和专利代理师协作的证据材料。

## 一、目标

代码抽取阶段应尽量回答：

1. 技术方案在代码中是否已经实现
2. 由哪些模块、接口、配置、状态流转共同支撑
3. 哪些技术特征有直接代码证据，哪些只是归纳实现
4. 哪些内容仍需研发补充，不能直接写成既有实现

## 二、抽取顺序

建议按以下顺序展开：

1. 以候选方案为单位建立映射，不要混写多个发明点
2. 先找入口、核心服务、调度逻辑和关键调用链
3. 再找关键数据结构、状态流转、缓存、鉴权、回退和异常处理
4. 最后提炼可写入交底书或专利初稿的技术特征

## 三、证据分级

| 级别 | 含义 | 使用要求 |
|------|------|----------|
| A 级证据 | 代码、配置、文档中有直接实现依据 | 可直接写入正文 |
| B 级证据 | 需要结合多个文件或运行逻辑做合理归纳 | 可以写入，但应注明“基于实现归纳” |
| C 级证据 | 当前仓库中未找到实现依据，只是推测或未来方向 | 不写成既有实现，应标为“待研发确认/可选实施例” |

### 起草层使用规则

1. A 级证据特征可进入独立权利要求骨架
2. B 级证据特征优先进入从属权利要求、优选实施例或“基于实现归纳”的说明
3. C 级证据特征只能进入“待研发确认”“替代实施例”或风险提示，不能写成既有实现

## 四、每条技术特征必须回答的问题

1. 该特征对应的技术问题是什么
2. 哪个模块、类、函数、接口或配置承担该特征
3. 关键调用链和数据流转是什么
4. 与常规实现相比，差异点在哪里
5. 是否足以支撑技术效果
6. 是否需要研发补充参数、阈值、条件或边界说明

## 五、引用要求

- 优先引用代码仓库中的相对路径
- 对关键实现注明模块、类、函数、接口、配置项或任务流名称
- 不大量粘贴代码，重点写清楚“代码做了什么”
- 没有找到依据时明确写“未在当前代码中定位到直接证据”

## 六、不宜直接写成专利特征的内容

- 纯业务规则、运营策略、文案逻辑
- 通用 CRUD 流程
- 仅是开发框架默认能力
- 完全来自第三方库或平台的固有机制
- 无法对应具体技术效果的常规工程做法

## 七、标准产出

建议形成以下文档：

1. `03-方案代码证据映射-<方案名>.md`
2. `04-技术交底书-<方案名>.md`
3. `04A-权利要求布局卡-<方案名>.md`
4. `04B-权利要求-代码证据矩阵-<方案名>.md`
5. `05-发明专利初稿-<方案名>.md`
6. `05A-发明专利初稿自检表-<方案名>.md`
7. `06-研发补充问题清单.md`

其中 `03-方案代码证据映射-<方案名>.md` 为核心前置产物。

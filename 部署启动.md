用户是直接访问 Frontend
访问地址就是脚本输出的前端端口，例如 3000 或回退到 3001，见 dev-no-nginx.sh:192。

脚本在启动前端时，直接注入了后端地址：
dev-no-nginx.sh:178
dev-no-nginx.sh:179

所以浏览器会直接请求：

# lead_agent config 
入口是 LangGraph 图注册
在 langgraph.json:9 里把 lead_agent 指到 agent.py:265 的 make_lead_agent。
LangGraph 运行这个图时会把本次 run 的 config 传给它。

这个项目里 config 的一个明确构造点
在 client.py:170 的 _get_runnable_config 会组装 RunnableConfig，包含 configurable 里的 thread_id、model_name、thinking_enabled、is_plan_mode、subagent_enabled，随后在 client.py:306 调用时传入。

是否“预定义”
分两层：

RunnableConfig 这个类型是框架预定义的（LangChain/LangGraph 类型）。
你在 configurable 里放哪些键（比如 is_plan_mode）是项目自己约定的，并不是框架强制固定字段。make_lead_agent 只是读取这些键（见 agent.py:270 和 agent.py:275）。
还有默认值来源
渠道管理器里有默认 run config/context，见 manager.py:19 到 manager.py:26。
所以最终是“默认值 + 调用方传入”合并后，传给 make_lead_agent。




# 文件上传路径
当前实现是“自动把上传文件信息（文件名/路径）放进上下文”，而图片真正进入多模态上下文需要经过 view_image 工具链路。

依据代码：

UploadsMiddleware 只是把上传文件转成 <uploaded_files> 文本块注入到用户消息
见 uploads_middleware.py:26 / uploads_middleware.py:120。
它提示用 read_file 读文件路径，不是直接注入图片二进制内容。
对应提示词也写了“自动列出上传文件 + 用 read_file 读取”：
prompt.py:246。

图片进入视觉上下文需要 view_image 工具
工具定义： view_image_tool.py:15。
它会读取图片并转 base64 写入状态。随后 ViewImageMiddleware 才把 image_url(data URL) 注入下一次模型调用：
view_image_middleware.py:121 / view_image_middleware.py:190。

## 本地 JSONL 日志怎么查看
日志目录由 DEERFLOW_RUN_EVENT_LOG_DIR 控制。

如果使用当前默认配置，目录是：

```bash
/nfsdat/home/akzhaoslm/deer-flow/logs/run_events
```

常用命令：

查看目录：

```bash
ls /nfsdat/home/akzhaoslm/deer-flow/logs/run_events
```

查看某个线程最近 50 行：

```bash
tail -n 50 /nfsdat/home/akzhaoslm/deer-flow/logs/run_events/<thread_id>.jsonl
```

筛选 agent 最终结果：

```bash
grep '"event": "agent.run.final"' /nfsdat/home/akzhaoslm/deer-flow/logs/run_events/<thread_id>.jsonl
```

筛选非流式结果：

```bash
grep '"event": "runs.wait.result"' /nfsdat/home/akzhaoslm/deer-flow/logs/run_events/<thread_id>.jsonl
```

筛选流式结束结果：

```bash
grep '"event": "runs.stream.final"' /nfsdat/home/akzhaoslm/deer-flow/logs/run_events/<thread_id>.jsonl
```

## LangSmith 怎么查看
1. 在 .env 中配置 LANGSMITH_TRACING、LANGSMITH_API_KEY、LANGSMITH_PROJECT。
2. 重启服务。
3. 发起一次真实对话、工具调用或上传操作。
4. 打开 LangSmith 控制台，进入 deer-flow 项目查看 traces。
5. 点击链接打开https://smith.langchain.com/

## 本地 JSONL 当前记录的事件
当前会记录以下事件：

- agent.run.final
- runs.wait.result
- runs.stream.start
- runs.stream.chunk
- runs.stream.error
- runs.stream.final

它们都会按 thread_id 分文件写入，适合排查：

- 文件上传相关问题
- Feishu 流式输出问题
- 非流式 wait 返回异常
- agent 最终状态与运行上下文


## 监控能力说明
当前项目支持两套监控方案：

1. LangSmith
用于查看模型调用链、trace、运行过程。

2. 本地 JSONL
用于按 thread_id 把关键运行事件落盘到本机，方便直接 tail、grep、排查问题。

两套可以同时开启，也可以分别单独开启。

## 当前使用的环境变量
LangSmith：

- LANGSMITH_TRACING
- LANGSMITH_API_KEY
- LANGSMITH_PROJECT
- LANGSMITH_ENDPOINT

本地 JSONL：

- DEERFLOW_RUN_EVENT_LOG_ENABLED
- DEERFLOW_RUN_EVENT_LOG_DIR

## 推荐配置示例
两套都开启：

```env
LANGSMITH_TRACING=true
LANGSMITH_API_KEY=your-langsmith-api-key
LANGSMITH_PROJECT=deer-flow
LANGSMITH_ENDPOINT=https://api.smith.langchain.com

DEERFLOW_RUN_EVENT_LOG_ENABLED=1
DEERFLOW_RUN_EVENT_LOG_DIR=/nfsdat/home/akzhaoslm/deer-flow/logs/run_events
```

只开 LangSmith：

```env
LANGSMITH_TRACING=true
DEERFLOW_RUN_EVENT_LOG_ENABLED=0
```

只开本地 JSONL：

```env
LANGSMITH_TRACING=false
DEERFLOW_RUN_EVENT_LOG_ENABLED=1
```

全部关闭：

```env
LANGSMITH_TRACING=false
DEERFLOW_RUN_EVENT_LOG_ENABLED=0
```

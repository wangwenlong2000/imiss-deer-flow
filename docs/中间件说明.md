# DeerFlow 中间件说明

本文说明 DeerFlow 后端 Lead Agent 的中间件体系：
- 具有哪些中间件
- 每个中间件的输入/输出
- 每个中间件的功能
- 中间件是怎样实现并串联起来的

## 1. 中间件总览

Lead Agent 的中间件在运行时由以下代码组装：
- `backend/packages/harness/deerflow/agents/lead_agent/agent.py`
- `backend/packages/harness/deerflow/agents/middlewares/tool_error_handling_middleware.py`

执行顺序（按实际组装顺序）：
1. `ThreadDataMiddleware`
2. `UploadsMiddleware`
3. `SandboxMiddleware`
4. `DanglingToolCallMiddleware`
5. `ToolErrorHandlingMiddleware`
6. `SummarizationMiddleware`（可选，配置开启时）
7. `TodoMiddleware`（可选，`is_plan_mode=true` 时）
8. `TitleMiddleware`
9. `MemoryMiddleware`
10. `RunHistoryMiddleware`
11. `ViewImageMiddleware`（可选，模型支持视觉时）
12. `SubagentLimitMiddleware`（可选，`subagent_enabled=true` 时）
13. `LoopDetectionMiddleware`
14. `ClarificationMiddleware`（固定放最后）

说明：
- 第 1~5 项由 `build_lead_runtime_middlewares()` 统一构建。
- 第 6~14 项由 `make_lead_agent()` 在运行时按配置追加。

## 2. 状态输入输出（ThreadState）

中间件主要读写 `ThreadState` 字段（定义在 `backend/packages/harness/deerflow/agents/thread_state.py`）：

- `messages`: 对话消息列表（所有中间件最核心输入）
- `sandbox`: 沙箱信息（如 `sandbox_id`）
- `thread_data`: 线程目录信息（workspace/uploads/outputs）
- `title`: 线程标题
- `artifacts`: 产物路径列表（带去重 reducer）
- `todos`: 计划模式任务列表
- `uploaded_files`: 本轮上传文件元数据
- `viewed_images`: 已查看图片数据（含 base64、mime，带合并 reducer）

可以把中间件理解成“流水线节点”：
- 输入：当前 state + runtime context（例如 `thread_id`）
- 输出：一个增量更新 dict（例如 `{"title": "..."}`）或 `None`
- 框架把这些增量合并回 state，传给下一个中间件

## 3. 各中间件详解

### 3.1 ThreadDataMiddleware

实现文件：
- `backend/packages/harness/deerflow/agents/middlewares/thread_data_middleware.py`

输入：
- `runtime.context.thread_id`

输出：
- `thread_data = { workspace_path, uploads_path, outputs_path }`

功能：
- 为当前 thread 计算（或创建）工作目录路径，保证后续上传、产物、沙箱工具有固定目录可用。

实现方式：
- 在 `before_agent()` 执行。
- 支持 `lazy_init`：
  - `True`：只计算路径，不立即创建目录。
  - `False`：立即创建目录。

### 3.2 UploadsMiddleware

实现文件：
- `backend/packages/harness/deerflow/agents/middlewares/uploads_middleware.py`

输入：
- 最后一条 `HumanMessage.additional_kwargs.files`
- 线程 uploads 目录中的历史文件

输出：
- 更新后的 `messages`（在最后一条用户消息前注入 `<uploaded_files>...</uploaded_files>` 块）
- `uploaded_files`（本轮新上传文件列表）

功能：
- 把“本轮上传文件 + 历史可用文件”告诉模型，便于模型直接调用 `read_file` 读取。

实现方式：
- 在 `before_agent()` 执行。
- 只处理最后一条用户消息。
- 对文件做基本安全过滤（例如文件名合法性、文件存在性）。

### 3.3 SandboxMiddleware

实现文件：
- `backend/packages/harness/deerflow/sandbox/middleware.py`

输入：
- `runtime.context.thread_id`
- 当前 `state.sandbox`

输出：
- 可能写入 `sandbox = { sandbox_id }`

功能：
- 为线程绑定沙箱环境，使 `bash/read_file/write_file` 等工具在隔离环境中执行。

实现方式：
- `before_agent()`：按需 `acquire` 沙箱（支持 lazy/eager）。
- `after_agent()`：释放当前运行使用的沙箱引用（由 provider 管理复用/生命周期）。

### 3.4 DanglingToolCallMiddleware

实现文件：
- `backend/packages/harness/deerflow/agents/middlewares/dangling_tool_call_middleware.py`

输入：
- `request.messages`

输出：
- patched `messages`（必要时插入占位 `ToolMessage`）

功能：
- 修复“AI 发起了 tool_calls，但对应 ToolMessage 缺失”的历史不完整问题（常见于中断场景），防止模型报格式错误。

实现方式：
- 用 `wrap_model_call()/awrap_model_call()` 包装模型调用。
- 在调用模型前扫描历史，发现 dangling tool call 就立刻补一条 `status="error"` 的 ToolMessage。

### 3.5 ToolErrorHandlingMiddleware

实现文件：
- `backend/packages/harness/deerflow/agents/middlewares/tool_error_handling_middleware.py`

输入：
- 工具调用请求 `ToolCallRequest`
- 工具执行异常

输出：
- 失败时返回 `ToolMessage(status="error")`，而不是直接抛异常中断整轮

功能：
- 将工具异常“降级”为可被模型理解的错误消息，提升鲁棒性。

实现方式：
- 用 `wrap_tool_call()/awrap_tool_call()` 包装工具执行。
- 捕获普通异常并构造标准化错误文本。
- 对 `GraphBubbleUp` 这类 LangGraph 控制流异常直接透传。

### 3.6 SummarizationMiddleware（可选）

实现位置：
- 创建逻辑在 `backend/packages/harness/deerflow/agents/lead_agent/agent.py` 的 `_create_summarization_middleware()`
- 实际类来自 `langchain.agents.middleware.SummarizationMiddleware`

输入：
- `messages`（上下文历史）
- summarization 配置（trigger/keep/model 等）

输出：
- 压缩后的对话上下文（由框架中间件内部完成）

功能：
- 当上下文接近限制时自动摘要旧消息，保留近期关键内容，降低 token 压力。

实现方式：
- 按配置动态创建并插入链路。
- 可配置触发条件、保留策略、摘要模型。

### 3.7 TodoMiddleware（可选）

实现文件：
- `backend/packages/harness/deerflow/agents/middlewares/todo_middleware.py`

输入：
- `state.todos`
- `messages`

输出：
- 必要时注入提醒消息（`todo_reminder`）

功能：
- 基于 LangChain 的 TodoListMiddleware 增强“上下文丢失自恢复”：
  - 如果摘要后原始 `write_todos` 调用被截断，模型可能忘记当前待办。
  - 此中间件会自动补提醒，保证任务追踪连续性。

实现方式：
- 继承 `TodoListMiddleware`。
- 在 `before_model()/abefore_model()` 检测：
  - state 有 todos
  - 但 messages 中看不到 `write_todos` 工具调用
  - 且尚未注入过 reminder
- 满足条件就插入一条系统提醒语义的人类消息。

### 3.8 TitleMiddleware

实现文件：
- `backend/packages/harness/deerflow/agents/middlewares/title_middleware.py`

输入：
- `messages`
- `title` 当前值
- 标题配置（是否启用、长度、模板）

输出：
- `title`

功能：
- 在“首轮完整对话”后自动生成线程标题。

实现方式：
- 在 `after_model()/aafter_model()` 判断触发条件：
  - 已有 1 条用户消息 + 至少 1 条助手消息
  - 且当前还没有 title
- 调轻量模型生成短标题并做长度/空值兜底。

### 3.9 MemoryMiddleware

实现文件：
- `backend/packages/harness/deerflow/agents/middlewares/memory_middleware.py`

输入：
- `messages`
- `runtime.context.thread_id`
- memory 配置

输出：
- 不直接改 state；把过滤后的对话加入 memory queue

功能：
- 把可长期记忆的信息异步入队（防抖批处理），用于后续记忆更新。

实现方式：
- 在 `after_agent()` 执行。
- 过滤消息：
  - 保留用户输入与最终 AI 回答
  - 去掉工具中间消息
  - 移除 `<uploaded_files>` 临时块，避免把会话级文件路径写入长期记忆

### 3.10 RunHistoryMiddleware

实现文件：
- `backend/packages/harness/deerflow/agents/middlewares/run_history_middleware.py`

输入：
- `messages`
- `artifacts/title/todos/uploaded_files/thread_data`
- `runtime.context`

输出：
- 不改 state；写入本地 JSONL 运行日志事件

功能：
- 持久化每轮最终执行快照，便于监控、排障、离线分析。

实现方式：
- 在 `after_agent()` 组装 payload，调用 `write_run_event_log()`。

### 3.11 ViewImageMiddleware（可选）

实现文件：
- `backend/packages/harness/deerflow/agents/middlewares/view_image_middleware.py`

输入：
- `messages`（尤其最后一条 AI 是否调用过 `view_image`）
- `viewed_images`

输出：
- 在下一次模型调用前注入一条包含图片内容块的人类消息

功能：
- 把 `view_image` 工具读取到的图片（含 base64）显式喂给视觉模型，让模型“看见”图片并继续推理。

实现方式：
- 在 `before_model()/abefore_model()` 执行。
- 仅在以下条件满足时注入：
  - 最近 AI 消息包含 `view_image` 工具调用
  - 该 AI 消息对应工具调用已全部完成
  - 尚未注入过同类图片说明消息

### 3.12 SubagentLimitMiddleware（可选）

实现文件：
- `backend/packages/harness/deerflow/agents/middlewares/subagent_limit_middleware.py`

输入：
- 最后一条 AI 消息的 `tool_calls`
- `max_concurrent_subagents`（运行时参数）

输出：
- 必要时替换最后一条 AI 消息，截断超量 `task` 工具调用

功能：
- 强制限制并发子代理调用数量，避免模型一次性发过多 `task` 导致资源爆发。

实现方式：
- 在 `after_model()/aafter_model()` 执行。
- 统计 `task` 调用，超出上限时保留前 N 个，删除其余。

### 3.13 LoopDetectionMiddleware

实现文件：
- `backend/packages/harness/deerflow/agents/middlewares/loop_detection_middleware.py`

输入：
- 最后一条 AI 消息 `tool_calls`
- 线程级历史哈希窗口

输出：
- 告警阶段：注入 `SystemMessage` 提醒停止重复调用
- 硬停止阶段：移除当前 AI 消息中的 `tool_calls` 并追加强制收尾文本

功能：
- 防止代理陷入“同工具同参数重复调用”的死循环。

实现方式：
- 对每组工具调用做稳定哈希（name+args，多集合无序一致）。
- 线程级滑动窗口统计重复次数：
  - 达到 `warn_threshold`：提示收尾
  - 达到 `hard_limit`：强制去除工具调用
- 内置 LRU 线程追踪，控制内存占用。

### 3.14 ClarificationMiddleware

实现文件：
- `backend/packages/harness/deerflow/agents/middlewares/clarification_middleware.py`

输入：
- `ask_clarification` 工具调用参数（question/context/options/type）

输出：
- `Command(update={messages:[ToolMessage]}, goto=END)`

功能：
- 拦截澄清问题并中断本轮执行，把问题直接抛给用户等待回复。

实现方式：
- 在 `wrap_tool_call()/awrap_tool_call()` 拦截工具调用。
- 非 `ask_clarification` 透传；命中后格式化问题并返回 `goto=END`。
- 由于它会中断执行，所以必须放在链路最后，避免影响其他中间件处理顺序。

## 4. 生命周期钩子与 I/O 模式

中间件主要通过 4 类钩子介入：

- `before_agent`：Agent 主逻辑开始前
  - 常见用途：准备 thread/sandbox 上下文
- `before_model` / `wrap_model_call`：模型调用前
  - 常见用途：修复消息格式、注入额外上下文
- `after_model`：模型返回后、工具执行前
  - 常见用途：裁剪 tool calls、循环检测
- `after_agent`：整轮执行完成后
  - 常见用途：异步记忆、日志落盘
- `wrap_tool_call`：工具调用前后包裹
  - 常见用途：工具异常降级、澄清中断

I/O 设计规律：
- 输入主要来自 `state.messages` 与 `runtime.context.thread_id`
- 输出尽量是“局部字段更新”而非重写整个 state
- 中断流程通过 LangGraph `Command(goto=END)` 完成

## 5. 中间件是怎样实现与扩展的

### 5.1 实现思路

1. 定义一个状态 schema（通常继承 `AgentState`，声明自己关心的字段）。
2. 继承 `AgentMiddleware[...]`。
3. 在合适钩子里读取 state/runtime，返回增量更新 dict。
4. 在 `make_lead_agent()` 的 `_build_middlewares()` 中按顺序挂载。

### 5.2 新增一个中间件的最小步骤

1. 在 `backend/packages/harness/deerflow/agents/middlewares/` 新建 `xxx_middleware.py`。
2. 实现类 `XxxMiddleware(AgentMiddleware[...])` 与所需钩子。
3. 在 `agent.py` 中引入并按正确顺序插入 `_build_middlewares()`。
4. 若中间件依赖配置，先在 config 层增加开关，再按条件挂载。
5. 补充对应测试（建议覆盖顺序、输入输出字段、异常路径）。

### 5.3 顺序设计原则（实践经验）

- 资源准备类（thread/sandbox）放前面。
- 消息修复类（dangling patch）要在模型读取历史前完成。
- 保护类（tool error handling、loop detection）应在关键路径上始终生效。
- 可能中断流程的中间件（clarification）放最后，避免截断前置逻辑。

## 6. 一句话总结

DeerFlow 的中间件体系本质上是一个“可配置、可中断、可恢复”的执行管线：
- 前段负责准备运行上下文
- 中段负责增强模型交互与工具安全
- 后段负责记忆与观测沉淀
从而在复杂多工具场景下保持稳定性、可追踪性与可扩展性。

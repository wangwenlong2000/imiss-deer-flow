# DeerFlow 启动命令与容器机制说明

本文面向第一次接触 DeerFlow 的同学，回答这些常见问题：

- make docker-init
- make docker-stop
- make docker-start
- make up
- make down

- `make config`、`make docker-init`、`make docker-start`、`make up`、`make down` 分别做什么
- `make` 和 `docker` 是什么关系
- 实际执行了哪些脚本
- 什么是热更新（hot reload）
- `Docker`、`Makefile`、`Dockerfile`、`docker-compose` 的关系
- 系统最终起几个容器、几个进程，彼此怎么协作
- 沙箱是不是单独容器，多会话时会不会残留很多容器

## 1. 五个命令分别做什么

## 1.1 `make config`

作用：生成本地配置文件。

实际入口：根目录 `Makefile` 的 `config` 目标，执行：

```bash
python ./scripts/configure.py
```

你可以理解为“初始化项目配置”。通常会生成/准备 `config.yaml`（以及相关配置文件）。

## 1.2 `make docker-init`

作用：预拉取沙箱镜像，减少第一次真正跑任务时的等待。

实际入口：

```bash
./scripts/docker.sh init
```

关键动作：

- 检查本地是否已有沙箱镜像
- 没有就执行 `docker pull enterprise-public-cn-beijing.cr.volces.com/vefaas-public/all-in-one-sandbox:latest`

这一步不启动业务容器，主要是“预热镜像缓存”。

## 1.3 `make docker-start`

作用：启动 Docker 开发环境（含开发态热更新）。

实际入口：

```bash
./scripts/docker.sh start
```

关键动作：

- 从 `config.yaml` 自动检测 sandbox 模式（`local` / `aio` / `provisioner`）
- 检查/补齐 `config.yaml` 和 `extensions_config.json`
- 调用开发编排文件启动容器：

```bash
docker compose -p deer-flow-dev -f docker/docker-compose-dev.yaml up --build -d --remove-orphans [services]
```

默认服务集：

- `frontend`
- `gateway`
- `langgraph`
- `nginx`

当检测到 `provisioner` 模式时会额外启动：

- `provisioner`

## 1.4 `make up`

作用：构建并启动生产环境容器。

实际入口：

```bash
./scripts/deploy.sh
```

关键动作：

- 准备生产环境变量（如 `DEER_FLOW_HOME`、`BETTER_AUTH_SECRET`）
- 检测 sandbox 模式
- 启动生产编排：

```bash
docker compose -p deer-flow -f docker/docker-compose.yaml up --build -d --remove-orphans ...
```

这条是“生产部署路径”，不是开发热更新路径。

## 1.5 `make down`

作用：停止并移除生产编排中的容器。

实际入口：

```bash
./scripts/deploy.sh down
```

本质调用：

```bash
docker compose -p deer-flow -f docker/docker-compose.yaml down
```

注意它对应的是生产项目名 `deer-flow`，和开发项目名 `deer-flow-dev` 不同。

## 2. `make` 和 `docker` 的关系

一句话：`make` 是任务入口与编排器，`docker` 是容器运行时。

- `make`：把一串命令封装成易记的目标（target），例如 `make docker-start`
- `docker`：真正执行镜像构建、容器启动、网络与卷挂载
- `docker compose`：定义并一次启动多容器系统

所以日常你敲的是 `make`，实际跑起来的是脚本里的 `docker compose`。

## 3. 命令背后执行了哪些脚本

从根 `Makefile` 到脚本链路可以简化为：

- `make config` -> `scripts/configure.py`
- `make docker-init` -> `scripts/docker.sh init`
- `make docker-start` -> `scripts/docker.sh start` -> `docker/docker-compose-dev.yaml`
- `make up` -> `scripts/deploy.sh` -> `docker/docker-compose.yaml`
- `make down` -> `scripts/deploy.sh down`

补充：

- 本地非 Docker 开发命令 `make dev` 走的是 `scripts/serve.sh --dev`
- 这是“宿主机多进程模式”，不是容器编排模式

## 4. 什么是热更新（Hot Reload）

热更新指：代码改动后服务自动重载，不需要你手工重启整套服务。

在 DeerFlow 中：

- 开发容器模式（`make docker-start`）
- 或本地开发模式（`make dev`）

都启用了开发态命令，例如：

- 前端：`pnpm run dev`
- Gateway：`uvicorn ... --reload`
- LangGraph：`langgraph dev`（开发服务器）

而生产模式（`make up`）追求稳定和性能，一般不启热更新。

### 4.1 热重载的底层原理（在本项目中的实现）

在 DeerFlow 的 Docker 开发模式（`make docker-start`）里，热重载不是单一机制，而是三层协同：

1. 文件可见层（Docker Volume 挂载）
  - 宿主机源码目录挂载到容器内路径
  - 你在本机编辑文件，容器内能立即看到新文件内容
  - 这一步只解决“代码变化是否进入容器”，本身不负责重启程序

2. 变更检测层（开发服务器 Watcher）
  - 容器内运行的是开发命令（dev server），而不是纯生产命令
  - dev server 持续监控文件变化
  - 检测到变化后触发重编译或重启子进程

3. 执行层（进程级重载，不是容器级重建）
  - 重启的是容器内应用进程/worker（例如 uvicorn reload worker）
  - 通常不会重建镜像，也不会重新创建容器
  - 因此反馈快，开发效率高

### 4.2 对应到 DeerFlow 三个核心服务

1. Frontend（Next.js）
  - 开发命令：`pnpm run dev`
  - 挂载：`frontend/src`、`frontend/public` 等目录
  - 机制：Next dev server 检测变更后热更新/增量编译
  - 补充：`WATCHPACK_POLLING=true` 用于增强容器场景下的文件变更检测稳定性

2. Gateway（FastAPI）
  - 开发命令：`uvicorn ... --reload`
  - 挂载：`../backend/:/app/backend/`
  - 机制：uvicorn reload 监控 Python 文件变化后重启应用进程

3. LangGraph
  - 开发命令：`langgraph dev`
  - 挂载：`../backend/:/app/backend/`
  - 机制：LangGraph 开发服务器处理代码变化并进行开发态重载

### 4.3 为什么说“不是容器自动重启”

- 容器本身是否重启，主要由 Docker/Compose 的 `restart` 策略决定（如 `unless-stopped`），它针对的是容器异常退出。
- 热重载针对的是“代码变化”，由应用层开发服务器处理。
- 所以常见现象是：容器 ID 不变，但容器内服务日志出现 reloading/restarting worker。

### 4.4 开发模式与生产模式差异

- 开发模式（`make docker-start`）：强调可迭代，使用源码挂载 + dev server 热重载。
- 生产模式（`make up`）：强调稳定，通常不启热重载（例如 LangGraph 使用 `--no-reload`，Gateway 使用多 worker 生产启动方式）。

## 5. `Docker`、`Makefile`、`Dockerfile`、`docker-compose` 的关系

可以这样记：

- `Makefile`：快捷按钮层（你输入的命令入口）
- `Dockerfile`：单个服务镜像配方（怎么从基础镜像构建出可运行环境）
- `docker-compose.yaml`：多服务拓扑图（哪些容器、端口、卷、网络、依赖）
- `Docker`：真正执行构建与运行的引擎

关系图（逻辑）：

1. 你执行 `make up`
2. `Makefile` 调脚本 `scripts/deploy.sh`
3. 脚本调用 `docker compose`
4. `docker compose` 按 `docker-compose.yaml` 读取每个服务的 `Dockerfile` 构建镜像并启动容器

## 6. 系统会启动几个容器、几个进程

需要分场景看。

## 6.1 `make docker-start`（开发容器模式）

常驻容器数量：

- 通常 4 个：`frontend`、`gateway`、`langgraph`、`nginx`
- provisioner 模式下 5 个：再加 `provisioner`

每个容器里至少有 1 个主进程；开发服务可能还会有重载子进程（如 `--reload` 监控进程）。

## 6.2 `make up`（生产容器模式）

常驻容器数量：

- 通常 4 个：`frontend`、`gateway`、`langgraph`、`nginx`
- provisioner 模式下 5 个：再加 `provisioner`

进程数特点：

- `gateway` 在生产 compose 里使用 `uvicorn ... --workers 2`，因此会有多 worker 进程
- 其他服务通常各自 1 个主进程

## 6.3 `make dev`（非 Docker，本机多进程）

容器数量：

- 0（核心服务直接在宿主机进程运行）

常驻进程（典型）：

- LangGraph
- Gateway
- Frontend
- Nginx

但如果你配置的是 AIO 沙箱，执行任务时仍可能额外拉起沙箱容器。

## 7. 都在 Docker 中运行吗

不是。

- `make docker-start` / `make up`：核心服务在 Docker 中运行
- `make dev`：核心服务在宿主机运行

沙箱是否使用 Docker，取决于 `config.yaml` 的 `sandbox.use`：

- `LocalSandboxProvider`：直接在宿主机执行，不起沙箱容器
- `AioSandboxProvider`：会用 Docker（或其他容器后端）创建沙箱实例

## 8. 沙箱是单独一个容器吗

在 AIO 模式下，沙箱是独立于核心服务容器的执行容器（或在 provisioner 模式下是独立 Pod）。

关键点：

- 沙箱 ID 对 thread 是确定性的（同一 thread 倾向复用同一沙箱）
- provider 有 warm pool（释放后容器可短期保留，便于下次复用）
- 有 idle timeout（默认 600 秒）后台清理闲置沙箱
- `replicas` 默认 3（软上限）：
  - 超过时优先淘汰 warm pool 中最旧沙箱
  - 如果全部都是活跃沙箱，可能暂时超过这个上限（不会强杀正在服务的沙箱）

## 9. 多个对话会不会在机器上留下很多容器

结论：可能会增加，但有控制机制，不会无限无序增长。

来源：

- 每个活跃 thread 可能对应一个活跃沙箱
- 释放后的沙箱可能先进入 warm pool

控制机制：

- `idle_timeout` 自动回收闲置沙箱
- `replicas` 控制常态容量（默认 3，软上限）
- `make stop` / `make docker-stop` 会调用清理脚本，按容器名前缀清理 `deer-flow-sandbox*`

如果你长期高并发、长会话同时活跃，沙箱数会相应上升；这是为了保证会话隔离与并发执行能力。

## 10. 一句话总结

- `make` 负责“调度命令”
- `docker` 负责“跑容器”
- `Makefile + shell 脚本 + docker-compose + Dockerfile` 共同构成启动体系
- 核心服务容器数通常 4（或 5，含 provisioner）
- 沙箱在 AIO/Provisioner 模式下是独立执行单元，会随会话动态创建并按策略清理

## 11. AioSandbox 的真实运行机制（重点）

你可以把 AioSandbox 理解为：

- `AioSandboxProvider` 在 LangGraph 进程内运行，负责管理 sandbox 生命周期
- 每个 `sandbox_id` 对应一个独立 sandbox 容器（或 provisioner 模式下的独立 Pod）
- `thread_id -> sandbox_id` 是确定性映射，同一对话线程通常会复用同一个 sandbox
- `release` 后不会立刻销毁，而是进入 warm pool，容器继续运行，便于下次快速复用

### 11.1 回答常见误解

不是“一个 LangGraph 容器里跑多个隔离 sandbox 实例”。

更准确地说是：

- LangGraph 进程里有一个 Provider（管理器）
- Provider 管理多个 sandbox 实例
- 每个 sandbox 实例背后是一个独立容器/Pod 端点

### 11.2 何时会新建容器

- 同一 `thread_id` 再次请求：优先复用已有 sandbox（活跃或 warm pool）
- 新的 `thread_id`：通常会新建新的 sandbox 容器/Pod
- 超过 `replicas`（软上限）时：优先淘汰 warm pool 中最老实例；活跃实例不会被强杀

### 11.3 何时会销毁容器

- 闲置超过 `idle_timeout`（默认 600 秒）会被后台回收
- 服务 shutdown 时统一销毁
- 容量淘汰时会销毁 warm pool 中的旧实例

## 13. LangGraph 容器内如何创建和管理 AioSandbox 容器

这部分回答 4 个高频问题。

### 13.1 LangGraph 在容器里运行时，能再创建 AioSandbox 容器吗

可以。

在开发容器编排里，`langgraph` 服务挂载了宿主机 Docker Socket：

- `/var/run/docker.sock:/var/run/docker.sock`

这意味着 LangGraph 进程内的 `AioSandboxProvider` 可以通过 Docker API 在宿主机创建/管理 sandbox 容器（DooD 模式）。

### 13.2 创建和管理代码在哪里

核心代码路径如下：

- Provider 生命周期管理（acquire / release / destroy / warm pool / replicas）：
  - `backend/packages/harness/deerflow/community/aio_sandbox/aio_sandbox_provider.py`
- 本地容器后端（真正执行 `docker run`、`docker stop`、`docker inspect`）：
  - `backend/packages/harness/deerflow/community/aio_sandbox/local_backend.py`
- 远程 provisioner 后端（走 HTTP 调 provisioner 创建/销毁 Pod）：
  - `backend/packages/harness/deerflow/community/aio_sandbox/remote_backend.py`
- 配置项定义（`image`、`replicas`、`idle_timeout`、`container_prefix` 等）：
  - `backend/packages/harness/deerflow/config/sandbox_config.py`

补充：sandbox 容器创建时的容器名规则在 `local_backend.py` 中，格式为：

- `{container_prefix}-{sandbox_id}`
- 默认前缀是 `deer-flow-sandbox`

### 13.3 AioSandbox 使用的都是同一个镜像吗

默认是同一个镜像：

- `enterprise-public-cn-beijing.cr.volces.com/vefaas-public/all-in-one-sandbox:latest`

但这不是硬编码唯一值，你可以在 `config.yaml` 的 `sandbox.image` 中自定义。后续新建的 sandbox 会使用当前配置的镜像。

### 13.4 我可以在 docker ps 里看到 AioSandbox 容器吗

可以。

如果当前有活跃或 warm pool 的 sandbox 容器，一般能在 `docker ps` 或 `docker ps -a` 看到类似名称：

- `deer-flow-sandbox-xxxxxxxx`

本机这次检查结果：

- 核心容器可见：`deer-flow-langgraph`、`deer-flow-gateway`、`deer-flow-frontend`
- 当前未发现 `deer-flow-sandbox-*` 容器（说明此刻尚未触发 AioSandbox，或已被回收）

## 14. Gateway 在系统工作中到底提供什么服务（详细示例）

一句话定位：Gateway 是 DeerFlow 的“平台服务 API 层（控制面）”，负责模型配置、MCP/Skill 管理、上传与制品、记忆等非推理核心能力。

对比：

- `LangGraph` 负责 Agent 推理与工具编排（Thought/Action/Observation）
- `Gateway` 负责支撑这些推理工作所需的“配置与资源服务”

### 14.1 在四个进程里，谁会和 Gateway 通信

主要是这两条：

- `frontend -> nginx -> gateway`
- `用户/脚本/SDK -> nginx -> gateway`

通常不会是：

- `langgraph` 在每次对话中都主动调用 `gateway`（两者更多是并行服务，共享配置文件/状态）

### 14.2 Gateway 的服务分类

Gateway 主要提供这几类 API：

- 模型服务：查询模型列表、模型详情
- MCP 配置服务：读取/更新 MCP server 配置
- Skill 服务：查询技能、启用/禁用技能
- Memory 服务：记忆数据的查询与管理
- 上传服务：文件上传、文档预处理与入库
- 制品服务：读取线程 artifacts（输出文件）
- 建议服务：生成 follow-up suggestions

### 14.3 端到端示例（按真实工作流）

#### 示例 A：前端打开“模型选择下拉框”

目标：用户在聊天页面选择要用的模型。

调用链路：

1. 前端请求 `/api/models`
2. `nginx` 按路由规则把 `/api/*` 转发到 `gateway`
3. `gateway` 读取模型配置并返回列表
4. 前端渲染可选模型

价值：把“模型可用性与配置查询”从 LangGraph 推理链路中解耦。

#### 示例 B：管理员在 UI 里启用一个 MCP Server

目标：开启某个 MCP 能力（如 github/filesystem）。

调用链路：

1. 前端提交配置更新到 `/api/mcp/...`
2. `gateway` 校验并写入 `extensions_config.json`
3. 后续 Agent 运行时，LangGraph 侧会读取最新配置并加载/刷新 MCP 工具

价值：MCP 的控制操作由 Gateway 统一对外，LangGraph 保持“执行面”职责。

#### 示例 C：用户上传 PDF/Excel 给 Agent 分析

目标：让 Agent 在后续对话中使用上传文件。

调用链路：

1. 前端上传到 `/api/threads/{id}/uploads`
2. `gateway` 处理 multipart 文件，落盘到线程目录
3. 文档类型可做预处理（例如转 Markdown）
4. 返回可访问路径（含虚拟路径/制品地址）给前端
5. 下一轮用户走 LangGraph 对话时，Agent 才会消费这些文件

价值：上传与文件管理是资源服务，交给 Gateway；Agent 只管消费。

#### 示例 D：用户下载 Agent 生成的结果文件

目标：下载报告、图表、导出文档等 artifacts。

调用链路：

1. 前端访问 `/api/threads/{id}/artifacts/...`
2. `gateway` 定位并返回文件流
3. 前端触发下载或在线预览

价值：统一文件访问入口与权限边界，不把文件服务混入推理服务。

#### 示例 E：记忆（Memory）查看与管理

目标：查看/调整系统记忆，提高个性化效果。

调用链路：

1. 前端调用 `/api/memory/...`
2. `gateway` 执行记忆查询/更新
3. 后续 LangGraph 在会话中读取记忆上下文

价值：记忆 CRUD 管理与对话推理解耦，便于审计和治理。

### 14.4 为什么要单独保留 Gateway，而不是都放到 LangGraph

主要收益：

- 职责分离：推理执行面（LangGraph） vs 管理控制面（Gateway）
- 维护清晰：配置、上传、文件、记忆等 API 不侵入 Agent 运行时
- 安全边界更清楚：资源访问与配置变更可集中治理
- 前端开发更稳定：大量 UI 管理接口不依赖推理流程可用性

### 14.5 用“前端操作”快速判断是不是会走 Gateway

经验法则：

- 如果是“聊一句话让 Agent 推理” -> 主要走 `langgraph`
- 如果是“改配置、查列表、传文件、拿文件、管记忆” -> 基本走 `gateway`

### 14.6 用户/客户端上传文件时，Frontend 与 Gateway 如何协作

先统一术语，避免混淆：

- Client（Browser）：用户本地的浏览器运行时（Chrome/Edge/Safari），负责文件选择、发起 HTTP 请求、展示页面。
- Frontend：部署在 `frontend` 服务里的 Next.js 应用代码（UI、状态管理、API 调用逻辑）。

关系是：

- 用户操作发生在 Browser
- Frontend 代码运行在 Browser 中并驱动请求

这个流程可以拆成两段：

- 上传入库阶段：`frontend -> nginx -> gateway`
- 对话消费阶段：`frontend -> nginx -> langgraph`

更精确地写成“执行主体”是：

- 上传入库阶段：`client(browser, 执行 frontend 代码) -> nginx -> gateway`
- 对话消费阶段：`client(browser, 执行 frontend 代码) -> nginx -> langgraph`

#### A. 上传入库阶段（Frontend 和 Gateway 的主协作）

用户在页面点击上传后，关键链路如下：

1. Client（Browser）中运行的 Frontend 代码组装请求
  - Browser 通过文件选择器拿到用户文件
  - Frontend 代码将文件组装为 `multipart/form-data`
  - 请求目标为：`/api/threads/{thread_id}/uploads`
  - Frontend 会携带当前线程 ID，用于线程隔离存储

2. Nginx 路由到 Gateway
  - 因为该路径属于 `/api/*`（非 `/api/langgraph/*`）
  - nginx 会把请求转发给 `gateway`

3. Gateway 接收并处理上传
  - 校验文件与路径安全性
  - 将文件写入线程隔离目录（uploads）
  - 对可转换文档执行预处理（例如转 Markdown）
  - 生成回包元信息：文件名、路径、虚拟路径、artifact 访问路径等

4. Client（Browser）接收结果，Frontend 更新 UI
  - Browser 收到响应
  - Frontend 展示“上传成功/失败”
  - Frontend 更新“当前线程已上传文件列表”
  - 后续用户可继续在同线程发起分析

#### B. 对话消费阶段（上传完成后，Gateway 不再主导）

上传成功后，用户再发送“请分析这个文件”时：

1. Client（Browser）中运行的 Frontend 将对话请求发往 `/api/langgraph/*`
2. nginx 将请求转发到 `langgraph`
3. LangGraph 在运行时读取线程上传文件信息并注入上下文
4. Agent 通过虚拟路径访问文件（如 `/mnt/user-data/uploads/...`）

所以要点是：

- Gateway 负责“文件进入系统”
- LangGraph 负责“对话里使用文件”

#### C. 常见失败场景里 Frontend 与 Gateway 的行为

1. 文件类型/内容不合法
  - Gateway 返回错误状态与错误信息
  - Frontend 展示失败提示，不会写入成功列表

2. 请求超时或网络中断
  - Frontend 侧显示上传失败或重试入口
  - Gateway 不会返回完整成功元信息

3. 文档预处理失败（例如转换异常）
  - Gateway 通常返回对应错误或降级结果（视实现）
  - Frontend 按返回状态展示“部分成功/失败”提示

#### D. 一句话记忆

- Client（Browser）是请求的实际发起方与响应接收方
- Frontend 是运行在 Browser 内的应用代码，负责交互逻辑与请求编排
- Gateway 负责上传校验、落盘和文件元信息返回
- LangGraph 负责后续推理时消费这些文件

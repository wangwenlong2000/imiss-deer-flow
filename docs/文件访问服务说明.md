# 文件访问服务说明

## 背景

这个文件访问服务的目标很直接：让 AI 在回复中给出一个文件路径后，用户可以在浏览器里直接点击链接打开文件。

可以把它理解成一条受限的“文件快递链路”：

1. AI 先输出一个指向宿主机文件的路径。
2. 前端识别这个路径，把它改写成一个浏览器可访问的 HTTP 地址。
3. Nginx 把这个 HTTP 请求转发给 gateway。
4. gateway 校验路径是否在允许范围内。
5. 校验通过后，gateway 把文件内容返回给浏览器。

这样做的好处是：用户拿到的是浏览器能打开的链接，但服务端仍然保留了路径白名单和越权校验，不会把整台机器的文件系统暴露出去。

## 允许访问的目录

当前只允许访问以下两类路径：

- /mnt/nas
- /home/anker/imiss-deer-flow/datasets

这两个目录都属于宿主机路径。

因为 gateway、langgraph、frontend、nginx 都运行在 Docker 容器中，所以要先把宿主机目录挂载进容器，容器内部才能看到这些文件。

## 实现方式

## 1. gateway 新增受限文件路由

在 gateway 中新增了一个接口：

- /api/host-files/{path:path}

实现文件位于：

- [backend/app/gateway/routers/host_files.py](backend/app/gateway/routers/host_files.py)

核心逻辑包括：

- 定义允许访问的根目录白名单
- 将请求中的路径解析为真实文件路径
- 拒绝白名单之外的路径
- 拒绝目录访问，只允许读取文件
- 再次解析真实路径，防止通过符号链接跳出白名单目录
- 根据文件类型返回文本、HTML 或二进制响应

换句话说，这个路由不是“给什么路径就读什么”，而是“只有命中白名单、并且没有逃逸风险的路径才允许读”。

## 2. gateway 注册新路由

为了让 FastAPI 真正提供这个接口，还把路由注册到了应用入口：

- [backend/app/gateway/app.py](backend/app/gateway/app.py)
- [backend/app/gateway/routers/__init__.py](backend/app/gateway/routers/__init__.py)

## 3. 前端自动把文件路径改写成可点击链接

AI 最自然的输出方式不是直接拼 API 地址，而是给出原始路径，例如：

- /mnt/nas/demo/report.pdf
- /home/anker/imiss-deer-flow/datasets/network-traffic/sample.csv

前端消息渲染层会自动识别这两类路径，并把它们改写成：

- /api/host-files/mnt/nas/demo/report.pdf
- /api/host-files/home/anker/imiss-deer-flow/datasets/network-traffic/sample.csv

相关实现位于：

- [frontend/src/components/workspace/messages/markdown-content.tsx](frontend/src/components/workspace/messages/markdown-content.tsx)

这样 AI 只需要输出“宿主机上的真实路径”，前端就会把它转换成浏览器可访问的链接。

## 4. Nginx 转发新接口

因为浏览器访问的是统一入口，所以 Nginx 也需要知道这个新路径该交给 gateway 处理。

相关配置位于：

- [docker/nginx/nginx.conf](docker/nginx/nginx.conf)
- [docker/nginx/nginx.local.conf](docker/nginx/nginx.local.conf)

新增的代理入口是：

- /api/host-files

## 5. Docker 容器挂载宿主机目录

为了让容器内部能访问宿主机文件，给 gateway 和 langgraph 容器增加了 datasets 目录挂载。

相关配置位于：

- [docker/docker-compose-dev.yaml](docker/docker-compose-dev.yaml)
- [docker/docker-compose.yaml](docker/docker-compose.yaml)

当前关键挂载包括：

- /mnt/nas:/mnt/nas:ro
- ${DEER_FLOW_ROOT}/datasets:/home/anker/imiss-deer-flow/datasets:ro
- ${DEER_FLOW_REPO_ROOT}/datasets:/home/anker/imiss-deer-flow/datasets:ro

其中 `:ro` 表示只读挂载，避免容器意外修改宿主机文件。

## 使用示例

## AI 输出示例

AI 可以直接输出这样的 Markdown 链接：

```md
[打开 NAS 报告](/mnt/nas/demo/report.pdf)
[打开数据集样本](/home/anker/imiss-deer-flow/datasets/network-traffic/sample.csv)
```

前端会自动把这些链接改写成 `/api/host-files/...` 形式。

## 浏览器最终访问示例

浏览器实际访问的地址会类似这样：

```text
/api/host-files/mnt/nas/demo/report.pdf
/api/host-files/home/anker/imiss-deer-flow/datasets/network-traffic/sample.csv
```

如果文件是文本、HTML、图片、PDF 或浏览器支持的格式，浏览器通常会直接打开。
如果是其他二进制格式，浏览器会按自身能力决定预览或下载。

## 安全限制

为了避免这个能力被滥用，当前实现做了几层限制：

- 只允许访问白名单目录
- 不允许通过路径穿越访问其他目录
- 不允许通过符号链接跳出白名单目录
- 只提供文件读取，不提供目录浏览
- 容器挂载为只读，降低误操作风险

## 验证方式

本次实现新增了聚焦测试：

- [backend/tests/test_host_files_router.py](backend/tests/test_host_files_router.py)

已验证的场景包括：

- 白名单目录内的文件可以正常访问
- 白名单外的文件会被拒绝
- 指向白名单外文件的符号链接会被拒绝

使用的测试命令如下：

```bash
cd backend
UV_CACHE_DIR=$PWD/.uv-cache UV_PROJECT_ENVIRONMENT=$PWD/.venv-local PYTHONPATH=. uv run pytest tests/test_host_files_router.py -v
```

## 部署或重启要求

代码改完后，需要让相关容器重新加载配置，否则新路由和新挂载不会生效。

至少需要重建或重启这些服务：

- gateway
- langgraph
- nginx

如果前端容器没有热更新到最新代码，也需要一起重启 frontend。

## 常见问题

## 为什么不直接让 AI 输出完整的 API 链接？

可以，但没有必要。

让 AI 只输出真实文件路径更自然，也更不容易把环境细节写死在提示词里。前端统一做改写，后续如果接口变更，只需要改一处。

## 为什么要在 gateway 做二次校验？

因为前端改写只能改善体验，不能作为安全边界。
真正的安全边界必须在服务端，所以 gateway 仍然要检查路径是否合法。

## 为什么 langgraph 容器也要挂载 datasets？

当前主要是为了让 AI 运行环境和文件访问服务看到同一套路径语义。这样 AI 在输出路径时，不会出现“模型看到的路径”和“gateway 能访问的路径”不一致的问题。

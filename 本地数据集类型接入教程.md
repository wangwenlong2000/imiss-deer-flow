# 本地数据集类型接入教程

这份文档说明如何在当前项目中新增一种“服务器本地数据集类型”，让系统既能识别用户上传文件，也能识别仓库里的本地数据集。

当前网络流量数据已经是一个完整参考实现。后续新增街景图像、轨迹数据或其他本地数据类型时，建议按同一套框架扩展。

## 一、先回答一个关键问题

### 后续新增数据集类型，还要不要继续修改主 prompt？

通常需要，但不一定每次都大改。

规则是：

- 如果你新增的数据类型只是复用现有“命名文件优先走专用 tool”的机制，一般不需要再改主 prompt
- 如果你新增的数据类型有新的识别方式、特殊优先级或新的文件语义，建议补一小段主 prompt 规则

当前主 prompt 已经支持这类通用逻辑：

- 有明确上传上下文时，上传文件优先
- 只有文件名、没有上传上下文时，本地数据集优先
- 双命中时再澄清
- 命名数据文件且存在专用 tool 时，先试专用 tool，再决定是否澄清

所以如果你后面新增街景图像、轨迹数据，并且也走“专用 backend tool”这条链，主 prompt 大概率只需要小修，不需要重构。

## 二、推荐的整体架构

新增一种本地数据集类型时，推荐按这五层扩展：

1. 主 prompt
2. skill
3. backend tool
4. 脚本
5. 数据目录与配置

建议始终保持这个原则：

- skill 负责触发和路由
- backend tool 负责宿主机执行
- 脚本负责具体处理
- sandbox 路径映射只做兼容增强，不做主执行链

### 2.1 领域级 resolver / tool 层

每类数据不要只靠 prompt 猜，而是应该有自己的 resolver 或 backend tool。

例如：

#### 网络流量

- 根目录：
  - `datasets/network-traffic/raw`
  - `datasets/network-traffic/processed`
- tool：
  - `network_traffic_prepare`
  - `network_traffic_analyze`

#### 街景图像

- 根目录：
  - `datasets/street-view/raw`
  - `datasets/street-view/processed`
- tool：
  - 例如未来的 `street_view_inspect`
  - `street_view_analyze`

这样用户说：

- `Geodo.pcap`
- `xxx.jpg`
- `某个街景样本`

系统应该先按领域去解析，再走该领域的专用 tool，而不是先全局 `ls`。

### 2.2 统一数据集注册层

这是更重要的长期方案。

如果后面要扩成十类本地数据集，不建议每一类都靠 skill 文案自己记目录和文件类型。更好的方式是增加一个统一的数据集注册配置，让系统知道：

- 每个 domain 叫什么
- 默认根目录有哪些
- 支持哪些扩展名
- 对应哪些 tool

例如可以设计成：

```yaml
datasets:
  - domain: network-traffic
    roots:
      - datasets/network-traffic/raw
      - datasets/network-traffic/processed
    file_types:
      - .pcap
      - .pcapng
      - .csv
      - .parquet
    tools:
      prepare: network_traffic_prepare
      analyze: network_traffic_analyze

  - domain: street-view
    roots:
      - datasets/street-view/raw
      - datasets/street-view/processed
    file_types:
      - .jpg
      - .png
      - .jpeg
      - .json
    tools:
      inspect: street_view_inspect
      analyze: street_view_analyze
```

这样主 agent 的规则就不再是：

- “死记哪个目录”

而是：

- “识别到文件类型或领域后，查数据集注册表，再走对应 tool”

当前项目还没有把这一层正式做成独立配置文件，但如果后续真的要扩到很多数据类型，建议优先往这个方向演进。

## 四、新增一种本地数据集类型时需要改哪些文件

以下按“街景图像”举例。

### 1. 设计数据目录

建议在 `datasets/` 下新增自己的根目录，例如：

```text
datasets/street-view/
  raw/
  processed/
  schema/
```

例如：

```text
datasets/street-view/raw/beijing/block-a/0001.jpg
datasets/street-view/raw/beijing/block-a/0001.json
datasets/street-view/processed/beijing/block-a/0001.caption.json
```

### 2. 新建 skill

建议新增：

```text
skills/custom/street-view-analysis/
  SKILL.md
  agents/
    openai.yaml
  scripts/
    resolve_local_file.py
    inspect_street_view.py
    analyze_street_view.py
```

最少要改：

- `skills/custom/street-view-analysis/SKILL.md`
- `skills/custom/street-view-analysis/agents/openai.yaml`

skill 要写清楚：

- 什么时候触发
- 用户只给文件名时也要先尝试本地解析
- 先用哪个 backend tool
- 不要默认先 `ls /mnt/user-data/...`

### 3. 新增 backend tool

如果街景图像需要专用处理，建议新增至少一个 tool：

例如：

- `street_view_inspect`
- `street_view_analyze`

文件放在：

```text
backend/packages/harness/deerflow/tools/builtins/
```

例如：

```text
backend/packages/harness/deerflow/tools/builtins/street_view_tool.py
```

同时要修改：

- [__init__.py](C:\Users\Administrator\Desktop\imiss-deer-flow-main\imiss-deer-flow-main\backend\packages\harness\deerflow\tools\builtins\__init__.py)

把新 tool 导出。

### 4. 在 config.yaml 注册新 tool

在 [config.yaml](C:\Users\Administrator\Desktop\imiss-deer-flow-main\imiss-deer-flow-main\config.yaml) 的 `tools:` 下新增，比如：

```yaml
  - name: street_view_analyze
    group: bash
    use: deerflow.tools.builtins:street_view_analyze_tool
```

如果有多个 tool，就都注册进去。

### 5. 编写宿主机脚本

如果是图像或其他本地数据，建议脚本放在 skill 目录的 `scripts/` 下面，由 backend tool 调用。

例如：

- `resolve_local_file.py`
- `inspect_street_view.py`
- `analyze_street_view.py`

建议遵守：

- tool 通过 `subprocess` 或直接 Python 调用脚本
- 脚本负责真正的数据解析
- skill 不直接承担执行能力

### 6. 必要时补 schema 或映射

如果是结构化数据，建议加：

- `datasets/<your-domain>/schema/README.md`
- `datasets/<your-domain>/schema/field_mapping.yaml`

如果不是表格型数据，也可以只写 README 和你自己的元数据规范。

### 7. 视需要增强 sandbox 路径映射

如果你希望通用 `ls` / `read_file` 在走偏时也能看到本地数据，可以在：

- [local_sandbox_provider.py](C:\Users\Administrator\Desktop\imiss-deer-flow-main\imiss-deer-flow-main\backend\packages\harness\deerflow\sandbox\local\local_sandbox_provider.py)

里增加映射。

例如网络流量已经映射了：

```text
/mnt/datasets/network-traffic -> datasets/network-traffic
```

街景图像后面也可以加：

```text
/mnt/datasets/street-view -> datasets/street-view
```

注意：

- 这只是兼容增强
- 主执行链仍然应该走 backend tool

## 五、主 prompt 什么时候需要修改

主要改这个文件：

- [prompt.py](C:\Users\Administrator\Desktop\imiss-deer-flow-main\imiss-deer-flow-main\backend\packages\harness\deerflow\agents\lead_agent\prompt.py)

建议只在这几种情况下修改：

1. 新类型文件需要新的全局识别规则
2. 现有“Clarification First”压过了专用 tool
3. 需要给“上传优先 / 本地优先 / 双命中再澄清”加新的领域例外

如果只是新增一个普通本地数据集类型，且它也遵守：

- 命名文件优先尝试专用 tool
- tool 失败后再澄清

那么主 prompt 一般只需要轻微补充，不需要重写。

## 六、建议的接入步骤

新增一个本地数据集类型时，建议按这个顺序：

1. 先规划 `datasets/<domain>/raw` 和 `processed`
2. 新建 `skills/custom/<domain>-analysis`
3. 新增 backend tool
4. 在 `config.yaml` 注册 tool
5. 写脚本并先在本地 CLI 下跑通
6. 再改 skill 文案
7. 最后视需要小修主 prompt
8. 重启 LangGraph 和 Gateway
9. 用真实问题做回归测试

## 八、如何判断该不该新增 tool

建议标准：

### 直接新增 backend tool

适合：

- 本地数据集在服务器上长期存在
- 需要宿主机访问
- 需要稳定执行专用脚本
- 不希望依赖 `/mnt/user-data/uploads`

### 只靠 skill

仅适合：

- 逻辑很轻
- 完全依赖已有通用工具
- 不需要宿主机专用访问

对你的场景，我建议：

- 网络流量：继续保留 backend tool
- 街景图像：也建议走 backend tool

## 九、回归测试建议

每新增一种数据集类型，至少测这三类问题：

1. 裸文件名

```text
使用 Outlook.pcap，先预处理，再分析目的端口分布。
```

2. 带自然语言前缀的文件名

```text
使用本地文件 Outlook.pcap，先预处理，再分析目的端口分布。
```

3. 如果存在上传文件，同名冲突测试

```text
使用 Gmail.flow.csv 做分析。
```

看系统是否会：

- 上传优先
- 无上传时本地优先
- 双命中时主动澄清

## 十、最后的建议

如果你后面真的要扩成十类本地数据集，建议下一阶段做一个统一的“数据集注册表”，统一描述：

- 数据类型名
- 默认 roots
- 支持扩展名
- 对应 skill
- 对应 tool

这样后面新增第 3、4、5 类时就不用每次都零散改规则。

当前阶段，最稳的策略仍然是：

- 主 prompt 定优先级
- skill 定领域路由
- backend tool 做宿主机执行
- sandbox 映射只做兼容增强

## 十一、如何开放通用沙盒只读访问某类本地数据

这一节解决的是：

- 通用 `ls`
- 通用 `read_file`

注意：

- 这只是兼容增强
- 主执行链仍然应该优先走专用 backend tool
- 建议默认只开放“只读访问”，不要直接开放通用写权限

```python
    "/mnt/datasets/network-traffic": ".../datasets/network-traffic",
    "/mnt/datasets/street-view": ".../datasets/street-view",

```

对应的路径别名也可以扩成：

```python
_DATASET_PATH_ALIASES = {

    "./datasets/network-traffic": "/mnt/datasets/network-traffic",
```

- `/mnt/datasets/street-view/raw`
- `/mnt/datasets/street-view/processed`

### 为什么建议只开放只读，而不是开放通用写权限

因为通用工具一旦能写这些目录，会带来几个问题：

- 模型更容易绕过专用 tool，直接在数据目录里随意改文件
- 权限边界变大
- 更难区分“原始数据”和“处理中间产物”
- 后续十类数据集一起开放时风险会持续放大

所以推荐策略是：

- `ls/read_file` 可以对白名单数据域做只读开放
- `write_file/str_replace` 仍然只允许线程私有目录
- 真正的数据处理、预处理、导出仍然交给专用 backend tool

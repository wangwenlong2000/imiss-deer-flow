# 本地数据集类型接入教程

这份文档说明：如何在当前项目里新增一种“服务器本地数据集类型”，并让系统同时支持：

- 用户上传文件
- 服务器本地数据集
- 严格 skill 工作流
- 原生 `read_file` + `bash` 链路
- 专用脚本分析

当前网络流量数据已经是一套完整参考实现。后续如果要新增街景图像、轨迹数据、日志样本或其他本地数据集类型，建议按同一套结构扩展。

## 当前项目的真实架构

现在推荐的主链是：

1. `lead_agent` prompt 做全局优先级约束
2. `SKILL.md` 做领域路由和严格执行规则
3. 系统原生 `read_file` 读取 skill 和小型参考文件
4. 系统原生 `bash` 执行领域脚本
5. 领域脚本完成真正的数据处理和分析

## 一、先回答一个关键问题

### 新增一种本地数据集类型后，还要不要继续修改主 prompt？

通常需要，但不一定每次都大改。

原则是：

- 如果新数据类型只是复用当前“上传优先、本地次之、命中 skill 后严格走脚本”的通用机制，通常只需要小改
- 如果新数据类型有新的优先级、新的文件语义，或者需要新的全局防误触规则，建议补一小段 prompt

当前项目的主 prompt 已经支持这些关键逻辑：

- 命中 skill 时优先读 skill
- 上传文件优先
- 无上传上下文时可回落到本地数据集
- 大型结构化数据文件不应直接全文 `read_file`
- 命中 skill 后不要优先自写临时代码

所以后续新增新的本地数据集类型，一般只需要：

- 补 skill
- 补脚本
- 补少量 prompt 规则

不需要重构整个 lead agent。

## 二、推荐的整体架构

新增一种本地数据集类型时，推荐按这五层扩展：

1. 主 prompt
2. skill
3. 领域脚本
4. 数据目录与 schema
5. sandbox 路径映射与只读开放

建议始终保持这个原则：

- prompt 负责全局优先级
- skill 负责触发与工作流
- 脚本负责真正执行
- sandbox 路径映射只做兼容增强，不承担主分析逻辑

## 四、新增一种本地数据集类型时需要改哪些文件

下面以“街景图像”为例。

### 1. 先设计数据目录

建议在 `datasets/` 下新增自己的根目录，例如：

```text
datasets/street-view/
  raw/
  processed/
  schema/
```

例如：

```text
datasets/street-view/raw/beijing/block-a/0001.jpg
datasets/street-view/raw/beijing/block-a/0001.json
datasets/street-view/processed/beijing/block-a/0001.caption.json
```

### 2. 新建 skill

建议新增：

```text
skills/custom/street-view-analysis/
  SKILL.md
  agents/
    openai.yaml
  scripts/
    resolve_local_file.py
    inspect_street_view.py
    analyze_street_view.py
```

至少要写清楚：

- 什么请求触发这个 skill
- 上传文件与本地数据集的优先级
- 允许哪些文件类型
- 哪些情况先 inspect
- 只能通过哪些脚本执行
- 明确禁止哪些兜底行为

### 3. 编写脚本

推荐把真正执行逻辑放在 skill 目录下的 `scripts/` 中，例如：

- `resolve_local_file.py`
- `inspect_street_view.py`
- `analyze_street_view.py`

建议遵守：

- `resolve` 负责输入解析
- `inspect` 负责结构确认
- `analyze` 负责领域分析
- skill 不直接承担数据处理

### 4. 视情况补 schema 或映射

如果是结构化数据，建议补：

- `datasets/<domain>/schema/README.md`
- `datasets/<domain>/schema/field_mapping.yaml`

如果是图像、文本片段、半结构化数据，也建议至少写一个 `README` 来定义字段和目录约定。

### 5. 必要时补 prompt 规则

主要修改：

- [prompt.py](C:\Users\Administrator\Desktop\imiss-deer-flow-main\imiss-deer-flow-main\backend\packages\harness\deerflow\agents\lead_agent\prompt.py)

建议只在这几种情况下改：

1. 新类型需要新的全局优先级
2. 现有 generic 行为会压过 skill
3. 需要新增“不要先全文读某类文件”的约束

## 五、如何开放通用 sandbox 对某类本地数据集的只读访问

这一层解决的是：

- 原生 `ls`
- 原生 `read_file`
- 偶尔偏离 skill 时的通用路径访问

也能看见某一类本地数据集，而不是一上来就权限拒绝。

注意：

- 这只是兼容增强
- 不应代替 skill 和脚本的主执行链
- 建议默认只开放只读访问

## 六、为什么建议只开放只读，而不是通用写权限

因为一旦通用工具能直接写这些目录，会带来几个问题：

- 模型更容易绕开 skill，直接乱改数据
- 原始数据与处理产物边界变模糊
- 权限面扩大
- 多个数据域一起开放后风险持续增加

所以推荐策略是：

- `ls` / `read_file` 对白名单数据域开放只读
- `write_file` / `str_replace` 仍然只用于线程私有工作区
- 真正的数据处理、预处理、导出继续交给领域脚本

## 七、推荐的接入顺序

新增一种本地数据集类型时，建议按这个顺序：

1. 规划 `datasets/<domain>/raw` 和 `processed`
2. 新建 `skills/custom/<domain>-analysis`
3. 写 `SKILL.md`
4. 写 `inspect` / `analyze` 脚本
5. 视需要补 schema
6. 补 sandbox 路径映射和只读开放
7. 小改主 prompt
8. 重启服务
9. 用真实问题做回归测试

## 八、回归测试建议

每新增一种数据集类型，至少测这三类问题：

### 1. 裸文件名

```text
分析 Outlook.pcap，先预处理，再分析目的端口分布。
```

### 2. 带自然语言前缀

```text
阅读本地数据集 Gmail.flow.csv，统计按字节数排名前 10 的源 IP。
```

### 3. 上传与本地同名冲突

```text
我刚上传了 Outlook.flow.csv，请优先阅读上传文件而不是本地同名数据集。
```

看系统是否做到：

- 上传优先
- 无上传时本地次之
- 同名冲突时不静默切换

## 九、长期建议

如果后面真的要扩成很多种本地数据集，建议下一阶段做一个统一的“数据集注册表”，统一描述：

- 数据集类型名
- roots
- 支持扩展名
- 对应 skill
- 默认 inspect/analyze 脚本

这样新增第 3、4、5 类数据集时，就不用每次零散改很多规则。

## 十、一句话总结

当前阶段最稳的策略仍然是：

- 主 prompt 定全局优先级
- skill 定领域路由和严格工作流
- 原生 `read_file` 只读 skill 和小参考文件
- 原生 `bash` 执行领域脚本
- sandbox 映射只做兼容增强，不承担主分析逻辑
